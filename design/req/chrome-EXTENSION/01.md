## 2. HCL-Like Output Format

### 2.1 Format Design Principles

```
DESIGN GOALS
════════════════════════════════════════════════
  ✓ Human readable and editable (like Terraform)
  ✓ Declarative — describes WHAT, not HOW
  ✓ Supports nested groups (child groups)
  ✓ Each service is a named block
  ✓ Comments supported (#, //)
  ✓ No quotes required for simple values
  ✓ Diff-friendly (one field per line)
```

### 2.2 Schema Definition

```hcl
# profiles/prod.cpf.hcl

profile "production-web-app" {
  version     = "1"
  created_at  = "2026-02-28"
  author      = "manh.dt"
  description = "Production environment — APAC region"

  # ── TOP-LEVEL GROUP ──────────────────────────────────────────────
  group "frontend" {
    label       = "Frontend Layer"
    region      = "ap-southeast-1"
    description = "CDN, storage, static assets"

    service "amazon-s3" {
      display_name    = "Amazon Simple Storage Service (S3)"
      service_url_key = "S3"
      configure_url   = "https://calculator.aws/#/createCalculator/S3"
      region = "ap-southeast-1"   # overrides group default

      # ── Section: S3 Standard ────────────────────────────────────
        section "s3_standard" {
        toggle_enabled = true
    dimension "storage" {
        quantity = 500
        unit     = "GB"
    }
    
    dimension "requests" {
        type     = "GET"
        quantity = 1000000
    }
    }

    service "amazon-cloudfront" {
      section "traffic" {
        field "Data transfer out to internet" = 1000  unit = "GB"
        field "HTTP requests"                 = 5000000
        field "HTTPS requests"                = 10000000
      }
    }

    # ── NESTED (CHILD) GROUP ────────────────────────────────────────
    group "frontend.cdn-edge" {
      label       = "Edge Caching"
      parent      = "frontend"
      description = "Regional edge PoPs"

      service "amazon-cloudfront" {
        label = "Edge - Asia PoPs"

        section "traffic" {
          field "Data transfer out to internet" = 200  unit = "GB"
        }
      }
    }
  }

  # ── SECOND TOP-LEVEL GROUP ───────────────────────────────────────
  group "backend" {
    label  = "Backend Services"
    region = "ap-southeast-1"

    service "amazon-rds" {
      section "database-instance" {
        field "Instance type"               = "db.r6g.large"
        field "Deployment option"           = "Multi-AZ"
        field "Storage type"                = "General Purpose SSD (gp3)"
        field "Storage amount"              = 100  unit = "GB"
        field "Additional backup storage"   = 50   unit = "GB"
      }

      section "data-transfer" {
        field "Data transferred"            = 10   unit = "GB"
      }
    }

    service "amazon-ec2" {
      section "instances" {
        field "Instance type"               = "t3.medium"
        field "Number of instances"         = 3
        field "Utilization"                 = 100  unit = "%"

        field "Operating system"            = "Linux"

        field "Volume type"                 = "General Purpose SSD (gp3)"
        field "Storage amount per instance" = 50   unit = "GB"
      }
    }

      label  = "Data Tier"
      parent = "backend"

      }
    }
  }
}
```

### 2.3 HCL Schema Grammar

```
GRAMMAR (EBNF-like)
══════════════════════════════════════════════════════════════════

profile  := "profile" STRING "{" profile_body "}"
profile_body := meta_field* group*

group    := "group" STRING "{" group_body "}"
group_body := (label | region | description | parent |
               service | group)*    ← groups can nest

service  := "service" STRING "{" service_body "}"
service_body := (label | region | section)*

section  := "section" STRING "{" section_body "}"
section_body := (field | "enabled" "=" BOOL)*

field    := "field" STRING "=" value ("unit" "=" STRING)?

value    := NUMBER | STRING | BOOL

STRING   := quoted "..." or bare word (no spaces)
NUMBER   := integer or decimal
BOOL     := true | false

BLOCK IDENTIFIER NAMING CONVENTION:
  service  → kebab-case  ("amazon-s3", "amazon-ec2")
  group    → dot-path    ("backend.data" signals parent=backend)
  section  → kebab-case  ("database-instance", "data-transfer")
  field    → raw AWS label as-is (preserves UI field names)
```

## 4. Functional Requirements

### 4.1 Content Script — DOM Capture

```
CAPTURE PIPELINE
══════════════════════════════════════════════════════════════════

  AWS Calculator DOM
        │
        ▼
  ┌─────────────────────────────────────────────────────────┐
  │  STEP 1: GROUP DETECTION                                │
  │  - Scan for group container elements                    │
  │  - Extract group names from headings                    │
  │  - Detect group nesting via DOM depth                   │
  └──────────────────────┬──────────────────────────────────┘
                         │
                         ▼
  ┌─────────────────────────────────────────────────────────┐
  │  STEP 2: SERVICE DETECTION (within each group)          │
  │  - Identify service cards/panels                        │
  │  - Normalize service names to catalog IDs               │
  │    ("Amazon Simple Storage Service (S3)" → "amazon-s3") │
  └──────────────────────┬──────────────────────────────────┘
                         │
                         ▼
  ┌─────────────────────────────────────────────────────────┐
  │  STEP 3: FIELD SCAN (within each service)               │
  │  - Scan all visible fields (text, number, dropdown,     │
  │    radio, toggle, slider)                               │
  │  - Detect section boundaries (collapsible headers)      │
  │  - Capture: label, value, unit, type, enabled/disabled  │
  └──────────────────────┬──────────────────────────────────┘
                         │
                         ▼
  ┌─────────────────────────────────────────────────────────┐
  │  STEP 4: TOGGLE PHASE CAPTURE                           │
  │  - Detect unchecked sections (S3 has 9 optional)        │
  │  - Record which sections are toggled ON/OFF             │
  │  - Capture hidden fields inside collapsed sections      │
  └──────────────────────┬──────────────────────────────────┘
                         │
                         ▼
  ┌─────────────────────────────────────────────────────────┐
  │  STEP 5: EMIT EVENT                                     │
  │  chrome.runtime.sendMessage({ type: "CAPTURE_COMPLETE", │
  │    payload: { groups, services, fields } })             │
  └─────────────────────────────────────────────────────────┘
```

### FR-01 — Group Capture
- Detect all service groups present on the AWS Calculator page
- Capture group names, order, and visual hierarchy
- Map AWS Calculator's group structure to `group {}` blocks in HCL
- Support nested groups: detect sub-groups from DOM depth, parent IDs, or indentation cues

### FR-02 — Service Capture
- Within each group, capture all service configurations
- Normalize AWS display names to canonical kebab-case IDs
- Associate each service with its parent group
- Support one service type appearing in multiple groups (with independent configs)

### FR-03 — Field Capture
- Capture all field types: number input, text input, dropdown/select, radio button, checkbox, toggle switch, slider, compound (value + unit)
- Record the field label exactly as shown in AWS Calculator UI
- Record current value and unit where applicable
- Record field state: enabled, disabled, hidden
- Group fields by their collapsible section header

### FR-04 — Toggle/Section State
- Detect which optional sections are expanded (enabled) vs collapsed (disabled)
- Record `enabled = false` for collapsed sections in HCL output
- When a section is expanded, capture all child fields within it

### FR-05 — Live Capture Mode
- Extension monitors DOM mutations in real time via MutationObserver
- When user changes a field value, extension updates its internal model immediately
- Side panel profile tree updates live without user needing to trigger manual capture

### FR-06 — Manual Snapshot
- User can trigger a full-page snapshot at any time via a "Capture Now" button
- Snapshot scans the entire page state regardless of what changed

### FR-07 — Profile Tree
- Display all captured groups and services as a collapsible tree
- Visual indicators: ✓ captured, ○ empty, ⚠ partial
- Click any node to jump to that section in the HCL preview
- Drag to reorder groups and services

### FR-08 — Group Management
- Add a new top-level group (named, with optional label/region)
- Add a child group under any existing group
- Move a service between groups via drag or context menu
- Rename, delete, or duplicate groups

### FR-09 — Live HCL Preview
- Right panel shows generated HCL in real time as capture happens
- Syntax highlighted (keywords, strings, numbers, comments in different colors)
- Active line tracks the currently selected tree node
- Editable — user can directly type changes, model updates accordingly

### FR-10 — Service Name Normalization
- Maintain a `field-map.json` catalog of AWS service display name → canonical ID
- When a new uncatalogued service is captured, prompt user to confirm/correct its ID
- Save user corrections back to field-map for future captures

## 6. Export Requirements

### FR-11 — Single Profile Export
- Export current profile as `{profile-name}.cpf.hcl` (plain text)
- Option to include shared variables file alongside

### FR-12 — Bundle Export (Multi-Profile)
- Select any combination of profiles to bundle
- Output: `{bundle-name}_{timestamp}.cpf.gz`
- Bundle is a gzip-compressed tar archive containing all selected `.cpf.hcl` files plus `manifest.json`

### FR-13 — Export Formats
```
FORMAT OPTIONS
════════════════════════════════════════════════
  .cpf.hcl          plain HCL, human-editable
  .cpf.hcl.gz       single-file gzip
  .cpf.gz           multi-file bundle (tar+gz)
  .json             raw internal model (debug)
  .yaml             YAML equivalent (optional)
```

### FR-14 — Import
- User can drag-and-drop a `.cpf.hcl` or `.cpf.gz` file onto the panel
- Extension loads the profile into the tree view
- Can then replay to AWS Calculator via Mode B runner (future integration point)

## 7. Non-Functional Requirements

### NFR-01 — Resilience to AWS Calculator Updates
```
SELECTOR STRATEGY — PRIORITY ORDER
════════════════════════════════════════════════════════════════

  1. ARIA roles + accessible names (most stable)
     [role="group"][aria-label="Amazon S3"]

  2. data-* attributes (stable if AWS uses them)
     [data-testid="service-s3"]

  3. Semantic HTML structure (moderately stable)
     form > fieldset > legend

  4. Text content matching (slow but resilient)
     label:contains("S3 Standard storage")

  5. CSS class names (NEVER use these as primary)
     .pricing-module-abc123  ← breaks every deploy

  Rule: any selector that breaks must degrade gracefully.
  Extension logs a warning, marks field as "unresolved",
  continues with rest of capture.
```

### NFR-02 — Performance
- MutationObserver must be debounced (150ms) to avoid flooding on rapid DOM changes
- Full page scan should complete in under 2 seconds for pages with up to 20 services
- HCL serialization should complete in under 100ms for profiles up to 200 fields

### NFR-03 — Permissions (Minimal)
```json
// manifest.json permissions
{
  "permissions": [
    "sidePanel",
    "storage",
    "downloads"
  ],
  "host_permissions": [
    "https://calculator.aws/*"
  ]
}
```
No broad host permissions, no remote code execution, no external API calls.

### NFR-04 — Manifest V3 Compliance
- Use service worker (not persistent background page)
- No remote-hosted scripts
- All logic bundled at install time

### NFR-05 — Profile Storage
- Profiles saved to `chrome.storage.local` during active session
- Max 10MB local storage, extension warns at 80% capacity
- User can export to disk at any time to free storage

## 9. Integration Contract with Mode B Runner

```
EXTENSION → NODE CLI HANDOFF
══════════════════════════════════════════════════════════════════

  Chrome Extension                  Node.js CLI (Mode B)
  ═════════════════                 ════════════════════
  
  Captures:                         Consumes:
  ┌─────────────────┐               ┌──────────────────┐
  │ prod.cpf.hcl    │──── import ──►│ cpf-parser.js    │
  └─────────────────┘               │  → profile model │
                                    │  → field mapper  │
                                    │  → browser auto  │
                                    └──────────────────┘

  The .cpf.hcl format is the CONTRACT between the two tools.
  Extension produces it. CLI runner consumes it.
  Both tools are independently useful.
```
